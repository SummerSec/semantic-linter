## 核心概念

**语义陷阱**：在人类日常语境中含义相近，但在大模型语义空间中激活范围（语义边界）存在巨大差异的词汇对。使用语义边界过宽的词汇，会导致大模型输出突破 Prompt 约束范围，产生幻觉。

**关键区分**：语义陷阱 ≠ 传统"用词不精确"。语义陷阱发生在指令已经很精确的前提下——定义了判定标准、提供了示例、设置了约束，但一个"关键名词"的不当选择就能让大模型认知范围被悄然扩大。

**核心机制**：词汇本身固有的语义宽度，能覆盖并突破 Prompt 建立的边界控制。

## 已知语义陷阱词汇对

### 高危词汇对（实验验证）

| ID | 窄边界词（推荐） | 宽边界词（谨慎） | 语义宽度差 | 失控场景 |
|----|----------------|----------------|-----------|---------|
| T01 | 漏洞 (Vulnerability) | 风险 (Risk) | 极高 | "风险"激活代码质量、架构设计、性能等大量超范围联想 |
| T02 | 检查 (Check) | 审查 (Review) | 高 | "审查"触发主观评价性意见，"检查"仅触发通过/不通过判定 |
| T03 | 列出 (List) | 描述 (Describe) | 高 | "描述"触发解释和评论，"列出"仅产出结构化条目 |
| T04 | 缺陷 (Defect) | 问题 (Issue) | 高 | "问题"范围极宽，改进建议都会被归类为"问题" |
| T05 | 总结 (Summarize) | 分析 (Analyze) | 中高 | "分析"触发推理和假设，"总结"仅提炼已有信息 |
| T06 | 要求 (Requirement) | 建议 (Suggestion) | 中高 | "建议"触发发散性思维，"要求"仅输出明确条件 |
| T07 | 错误 (Error) | 异常 (Anomaly) | 中 | "异常"边界模糊，轻微偏差都可能被纳入报告 |

## LLM 语义敏感度矩阵

二维框架：X轴=边界清晰度（高→低），Y轴=发散倾向（低→高）

```
            边界清晰度
            高 ←————————→ 低
        ┌─────────────────────┐
  低    │  漏洞  缺陷  错误    │  ← 安全区：语义收敛，输出可控
  ↑     │  检查  列出  要求    │
  发    │  验证  提取  匹配    │
  散    │───────────────────── │
  倾    │  总结  问题  审查    │
  向    │  建议  分析  风险    │  ← 危险区：语义发散，易失控
  ↓     │  异常  评估  描述    │
  高    └─────────────────────┘
```

**使用方法**：将待检测文本中的关键词定位到矩阵中，处于危险区的词汇需要重点关注。

## 判定示例

### 示例 1：检测到语义陷阱

**输入文本**（Skill 指令片段）：
```
请审查这段代码中的安全问题，只关注定义文件中的问题类型。
```

**检测结果**：
- `审查` → 宽边界词（T02），推荐替换为 `检查`
- `问题` → 宽边界词（T04），推荐替换为 `缺陷` 或 `漏洞`
- 风险等级：高
- 替换建议：`请检查这段代码中的安全漏洞，只关注定义文件中的漏洞类型。`

### 示例 2：检测到语义陷阱

**输入文本**：
```
分析用户行为数据，给出改善建议。
```

**检测结果**：
- `分析` → 宽边界词（T05），推荐替换为 `总结`
- `改善` → 宽边界词（T08），推荐替换为 `修复`（如针对具体问题）
- `建议` → 宽边界词（T06），推荐替换为 `要求`（如有明确标准）
- 风险等级：高
- 替换建议：视具体场景，如目的是找出具体问题：`总结用户行为数据中的异常模式，列出需要修复的具体缺陷。`

### 示例 3：未检测到语义陷阱

**输入文本**：
```
检查这段代码是否存在SQL注入漏洞，列出所有发现的缺陷。
```

**检测结果**：
- `检查` → 窄边界词，无需替换
- `漏洞` → 窄边界词，无需替换
- `列出` → 窄边界词，无需替换
- `缺陷` → 窄边界词，无需替换
- 风险等级：低
- 建议：当前用词语义边界清晰，无需调整。

### 示例 4：上下文相关判定

**输入文本**：
```
请评估这个方案的可行性。
```

**检测结果**：
- `评估` → 宽边界词，但此处作为独立任务目标（非约束条件中的关键名词），语义陷阱风险降低
- 风险等级：中
- 建议：如需收敛输出，替换为 `验证这个方案是否满足以下条件：[具体条件列表]`

## 边界锚定策略

当业务需求必须使用宽边界词时，采用以下策略锚定语义边界：

### 策略 1：前置否定清单

在定义中明确排除不属于评估范围的内容：

```markdown
## 本 Skill 定义的"风险"仅包括：
1. 类型A
2. 类型B

## 以下不属于评估范围（即使通常也被视为"风险"）：
- 代码质量问题
- 通用安全问题
- 架构设计问题
```

### 策略 2：输出格式硬约束

将输出设计为结构化填空题，消除塞入额外内容的空间：

```json
{
  "类型A判定": "无风险 | 低危 | 中危 | 高危",
  "类型B判定": "无风险 | 低危 | 中危 | 高危"
}
```

### 策略 3：反例强化

在示例中明确展示"虽然看起来像但不属于"的反例：

```markdown
### 反例：以下情况不属于本 Skill 定义的风险
代码中某参数未做格式校验。
判定：不在评估范围 → 默认无风险
```
